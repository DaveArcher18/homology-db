# Homology DB 0.0.1 corpus source pins

Date: 2026-07-11  
Status: ticket-07 source qualification; not an ingestion or licensing decision

## Result

The first corpus can be sourced reproducibly without treating a moving service
or an engine serialization as canonical:

- freeze the 2,000-record bulk reserve at **Stellar v6**, using 1,000 records
  from the exact-ten-vertex 2-manifold enumeration and 1,000 from the
  exact-ten-vertex 3-manifold enumeration; the manifest contract qualifies 960
  counted Models from that fixed reserve;
- use **polyDB only as a live overlap check**, never as the frozen source of
  those records;
- freeze the thirteen selected lens-space artifacts at one **simpcomp Git
  commit**, retaining each `.scb` blob but decoding it to neutral facet data;
- freeze **SageMath 10.9** for the selected `RP^2`, `RP^4`, `RP^5`, `RP^6`,
  and `CP^2` recipes, and hash each neutral output after materialization; and
- keep **HAP** as a pinned independent source/oracle. Its unmaterialized Dehn
  surgery recipes and **Regina** generalized triangulations do not enter the
  counted 0.0.1 corpus.

This note distinguishes three things deliberately:

1. a **source lock**, whose upstream bytes or recipe are immutable;
2. a **materialized neutral Model artifact**, which still has to be produced
   and validated by this repository; and
3. a **counted Model**, which additionally passes the model-boundary,
   distinctness, computation, provenance, and release-ledger gates.

An upstream row or constructor is not yet a counted Model.

## Verification method and limits

All factual source claims below were checked against official repositories,
official APIs, official documentation, or the source-owning research pages.
The Stellar files were downloaded directly from the versioned Zenodo record;
the SHA-256 values below were computed over those downloaded bytes, while the
MD5 values are the checksums published by Zenodo. The simpcomp and HAP trees
were checked out at the stated commits, and the listed per-file SHA-256 values
were computed locally.

No working Sage or GAP runtime was available in this investigation. Therefore:

- Sage and simpcomp recipe outputs have **not** yet been materialized;
- the simpcomp `.scb` objects have **not** yet been decoded in this repository;
- cached upstream names and homology have not been promoted to database facts;
  and
- all neutral-output hashes, decoder round trips, owned homology runs, and
  cross-engine comparisons remain release work.

Direct enumeration verified that the ID sequence of each main file is exactly
the sequence in both of its sidecars. The SHA-256 of the newline-separated ID
sequence is:

  `15c4366b0fcd2c4469bc379aea500047f9b79da84dc0c3237bb81ba48b8ecd9f`;
- 3-manifold trio:
  `ca703ef99c942ba1da1fc5664b78a0494d78905cfa1372a1780b5ec2916031ba`.

The observed type distributions are also useful selection guards:

- the 2-manifold data have 11 exact type strings, from 233 spheres and 1,210
  projective planes through orientable genus 3 and nonorientable genus 7;
- the 3-manifold data contain 247,882 `S^3`, 518 `S^2 x S^1`, and 615
  `S^2 twist S^1` rows.

Those 3-manifold counts agree with Lutz's source-owning
[complete-enumeration paper](https://arxiv.org/abs/math/0604018). The 2-manifold
enumeration is tied to the source page's table of 42,426 ten-vertex surfaces
and the Sulanke--Lutz
[isomorphism-free enumeration](https://arxiv.org/abs/math/0610022). This is
strong upstream distinctness evidence, not permission to skip parser-level
duplicate and targeted isomorphism checks.

### Deterministic reserve and 960-Model qualification rule

Within each main file, hash the exact raw record bytes and order by
`(raw_record_sha256, upstream_id)` bytewise. Ranks `0..499` form that
dimension's finite-simplicial reserve and ranks `500..999` its disjoint
finite-regular-CW reserve. Preserve every reserve record and its two ID-joined
sidecar rows separately.

For each of the four dimension/format strata, scan the fixed 500-record reserve
and admit the first 240 candidates that pass decoding, round-trip,
presentation-isomorphism, regularity where applicable, owned computation, and
release-ledger checks. A failed stratum fails the release instead of expanding
the reserve. The resulting 960 Models are 480 simplicial and 480 regular-CW
face-poset carriers. Each source record is assigned to one admitted class; a
facet document and its lossless closure-poset encoding never count twice.

### Stellar update and interpretation risks

- The v6 record explicitly changed 26 line endings in the small-valence file
  and repaired a missing record separator in `2_manifolds.txt`; importer tests
  must be version-specific.
- Type labels such as `S^3/Q` are not sufficient conceptual-space identities.
- The tuple notation in homology sidecars is convention-sensitive. Store it as
  an attributed imported assertion, then compare it with owned computation.
- Preserve DOI, record number, file key, full-file hashes, upstream ID, raw
  record bytes/hash, decoder version, and neutral artifact hash.

## polyDB: overlap check, not release input

The live
[`Manifolds.DIM2_3` collection](https://polydb.org/rest/current/collection/Manifolds/DIM2_3)
attributes the data to Frank Lutz and Constantin Fischer and points back to the
Stellar lineage. Its official
[schema](https://polydb.org/rest/current/schema/Manifolds/DIM2_3) requires
`_type = topaz::SimplicialComplex`, `FACETS`, `F_VECTOR`, and `HOMOLOGY`, so a
record is reconstructible rather than invariant-only.

A direct read of `manifold_2_10_1` showed the same facets as Stellar after the
transparent label shift from Stellar's `1..10` to polyDB's `0..9`. Returned
records carried `_polyDB.version = "2.1"`, but the documented REST base is
`/rest/current/`, and the
[REST guide](https://polymake.org/doku.php/polydb/rest/intro) does not promise an
immutable dataset snapshot. A tested `/rest/2.1/` count URL did not provide a
usable frozen endpoint.

Consequently polyDB is not a seventh source file and cannot inflate counts.
For a selected overlap, retain the exact raw JSON, retrieval timestamp, raw
SHA-256, collection version metadata, and `_id`; compare its normalized facets
and homology assertion to the frozen Stellar record. Never make the live
response a clean-build dependency.

## simpcomp: exact artifact lock

### Repository and decoder pin

Use commit
[`aff7cf2bcef0828fb4d02296cf276b777705274b`](https://github.com/simpcomp-team/simpcomp/tree/aff7cf2bcef0828fb4d02296cf276b777705274b).
The checked tree contains exactly 648 `.scb` files: 422 under `transitive`, 181
under `manifolds`, and 45 under `pseudomanifolds`. These are source-artifact
counts, not release counts.

The package's pinned
[`PackageInfo.g`](https://github.com/simpcomp-team/simpcomp/blob/aff7cf2bcef0828fb4d02296cf276b777705274b/PackageInfo.g)
still declares version 2.1.14 dated 2022 even though this commit is from 2025;
therefore the commit, not the package version string, is the content pin. It
requires GAP >= 4.8 and the GAP `io` package >= 3.0.

The pinned
[`SCLoad` implementation](https://github.com/simpcomp-team/simpcomp/blob/aff7cf2bcef0828fb4d02296cf276b777705274b/lib/io.gi#L718-L783)
uses `IO_Unpickle`; `.scb` is an engine-native binary object, not a neutral
facet format. The binary `complexes.idxb` is also pickled. The
[`SCLibLoad` implementation](https://github.com/simpcomp-team/simpcomp/blob/aff7cf2bcef0828fb4d02296cf276b777705274b/lib/lib.gi#L190-L226)
shows that numeric library IDs index file paths, while index rebuilding sorts
entries by f-vector. A stable source key is therefore commit + relative path +
raw SHA-256, not the numeric library ID.

## SageMath 10.9: selected RP and CP2 recipes

Use release
[`10.9`](https://github.com/sagemath/sage/releases/tag/10.9), commit
[`686dc1a8d420c2e0aabadd4f602d9a0aa4690c50`](https://github.com/sagemath/sage/tree/686dc1a8d420c2e0aabadd4f602d9a0aa4690c50).
Record the full qualified callable, canonical JSON arguments, Sage commit,
runtime/dependency lock, and normalized output hash for every generated Model.
Never retain `.sobj` as the canonical artifact.

The pinned
[`simplicial_complex_examples.py`](https://github.com/sagemath/sage/blob/686dc1a8d420c2e0aabadd4f602d9a0aa4690c50/src/sage/topology/simplicial_complex_examples.py)
supports the conservative route:

| Conceptual space | Pinned callable | Source-backed output shape | Qualification |
|---|---|---|---|
| `RP^2` | `simplicial_complexes.RealProjectivePlane()` | literal 6-vertex, 10-facet abstract simplicial complex | safe after neutral export |
| `RP^4` | `simplicial_complexes.RealProjectiveSpace(4)` | literal 16-vertex, 150-facet abstract simplicial complex | safe after neutral export |
| `RP^5` | `simplicial_complexes.RealProjectiveSpace(5)` | deterministic quotient construction; 63 vertices and 2,520 facets | safe recipe; run and hash |
| `RP^6` | `simplicial_complexes.RealProjectiveSpace(6)` | same construction; 127 vertices and 20,160 facets | safe recipe; run, measure, and hash |
| `CP^2` | `simplicial_complexes.ComplexProjectivePlane()` | literal 9-vertex, 36-facet abstract simplicial complex | safe after neutral export |

For `n >= 5`, the projective-space source constructs the barycentric
subdivision of the boundary of an `(n+1)`-simplex and quotients the antipodal
action. The listed facet counts are a direct consequence of the pinned
algorithm: `(n+2)! / 2`. Normalize the returned facet set explicitly because
source-level set iteration is not a persistence format.

The same source defines `RP^7`, `RP^8`, and `RP^9`, but the implied top-facet
counts are 181,440; 1,814,400; and 19,958,400. They were not executed here and
are outside the conservative 0.0.1 slice until measured import and owned-SNF
evidence exists.

Sage's pinned
[`ComplexProjectiveSpace(3)` and `(4)`](https://github.com/sagemath/sage/blob/686dc1a8d420c2e0aabadd4f602d9a0aa4690c50/src/sage/topology/simplicial_set_examples.py#L533-L631)
return finite **simplicial sets**, not abstract simplicial complexes. The
underlying Kenzo outputs contain 84 and 1,189 nondegenerate simplices,
respectively, but finite simplicial sets are outside the 0.0.1 presentation
boundary. Therefore `CP^3` and `CP^4` are explicit gaps, not silently converted
Models.

For the required genuinely nonsimplicial regular-CW slice, the pinned
[`cubical_complex.py`](https://github.com/sagemath/sage/blob/686dc1a8d420c2e0aabadd4f602d9a0aa4690c50/src/sage/topology/cubical_complex.py)
provides deterministic elementary-cube examples including spheres, a torus,
`RP^2`, the Klein bottle, closed surfaces, and cubes. Sage does not supply the
project's Regularity witness. Export each cube as its integer interval tuple,
run the owned strict face-to-face/closure checker, and only then translate the
same Model to its complete Cell-closure poset. Random Sage constructors are
excluded.

## HAP 1.76: source and oracle only

Pin tag `v1.76`, commit
[`6677964ab9ca90ecfff0cc920bcb0be1e0ae130f`](https://github.com/gap-packages/hap/tree/6677964ab9ca90ecfff0cc920bcb0be1e0ae130f).
Its
[`PackageInfo.g`](https://github.com/gap-packages/hap/blob/6677964ab9ca90ecfff0cc920bcb0be1e0ae130f/PackageInfo.g)
declares HAP 1.76, GAP >= 4.12, and several required GAP packages; pinning HAP
alone is not a reproducible runtime lock.

The pinned
[`manifolds.gi`](https://github.com/gap-packages/hap/blob/6677964ab9ca90ecfff0cc920bcb0be1e0ae130f/lib/Manifolds/manifolds.gi)
contains literal simplicial facet recipes for a torus, `RP^2`, `CP^2`, and K3.
These are useful independent fixtures. HAP implements
[`ThreeManifoldViaDehnSurgery`](https://github.com/gap-packages/hap/blob/6677964ab9ca90ecfff0cc920bcb0be1e0ae130f/lib/Manifolds/dehn.gi),
and its official
[three-manifold tutorial](https://gap-packages.github.io/hap/tutorial/chap4_mj.html)
describes lens-space recipes for coprime surgery coefficients.

Those Dehn-surgery outputs are not counted inputs in the conservative
manifest. They require a fully pinned GAP environment, deterministic
materialization, neutral export, and either an owned verified regular
subdivision or a separately admitted simplicial Model. An upstream `Regular
CW-complex` label is not the repository's Regularity witness. In particular,
the recipe is a possible route to lens spaces not present in the selected
simpcomp files, but it does not close those gaps yet.

## Regina: no counted 0.0.1 input

Regina's official
[triangulation documentation](https://regina-normal.github.io/docs/triangulations.html)
uses top-dimensional simplices with facet pairings; faces may be identified.
That is not automatically an abstract simplicial complex or the admitted
witness-certified regular-CW presentation. Regina may be pinned later as an
independent 3-manifold oracle, but no Regina census record belongs in the
0.0.1 counted corpus manifest.

## Explicit gaps and unknowns

1. **No neutral output hashes yet:** Sage recipes and all simpcomp blobs still
   require pinned-runtime materialization and canonical export.
2. **No decoder proof yet:** the exact simpcomp paths and raw hashes are known,
   but the `.scb` decoder has not been run locally and no round-trip fixture
   exists in this repository.
3. **`CP^3` and `CP^4` are not admitted:** the audited Sage artifacts are
   finite simplicial sets. No pinned admitted triangulation was established.
4. **`RP^7`--`RP^9` are not release-qualified:** source code defines them, but
   their rapidly growing facet counts have no measured import/computation
   evidence here.
5. **Lens spaces beyond the thirteen blobs are recipes, not artifacts:** in
   particular, simpcomp's constructor excludes arbitrary even `p`, and a HAP
   surgery recipe is not a counted substitute until materialized and checked.
6. **Upstream identity and homology remain assertions:** filenames, cached
   names, type sidecars, and cached homology never bypass reviewed
   `model_of` evidence or owned computation.
7. **Source counts are not ledger counts:** the 960 qualified Stellar targets and 13
   simpcomp blobs must still pass parsing, distinctness, one-presentation-class
   accounting, and the full G10 computation path.

## Minimum manifest fields forced by this audit

Every selected input record should carry at least:

- source system and source role;
- immutable source coordinate: DOI record or Git commit;
- upstream relative path/file key and upstream ID where applicable;
- retrieval timestamp;
- full source-file hash and selected raw-record/blob SHA-256;
- source format and decoder version;
- generator callable plus canonical arguments for recipes;
- complete runtime/dependency lock for executed recipes;
- normalized neutral artifact schema version and SHA-256;
- imported name/type/homology as separate attributed assertions;
- owned validation/computation run IDs; and
- duplicate/model-relation evidence without silent identity merging.

With those conditions, the source plan is reproducible and attainable. It is
not yet evidence that ingestion, distinctness, homology computation, or the
release corpus gates pass.
